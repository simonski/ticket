# UX Review

**Score: 78/100** (was 79)

## What is being assessed

Consistency and usability across the web UI and terminal-facing views, including visual hierarchy, validation feedback, accessibility basics, interaction clarity, and whether workflow state is communicated cleanly.

## Methodology

Reviewed `web/static/index.html`, the CLI output changes in `cmd/tk/printer.go`, and the previous UX report baseline, focusing on navigation, feedback, accessibility, and state visibility.

## Findings

### Passing checks
- **The CLI ticket-detail view now presents child state more clearly** — child rows remain visible and are visually de-emphasised/promoted by status (`cmd/tk/printer.go:262-307`)
- **`tk get` now shows child counts directly** — users can see child totals/open/closed without manual scanning (`cmd/tk/cmd_ticket.go:644-658`)
- **The web UI still ships as a single embedded app with consistent styling primitives** — the UI remains centrally defined in `web/static/index.html`
- **Mobile viewport support remains enabled** — the SPA keeps responsive viewport metadata in the main HTML

### Issues found
| Finding | Severity | Location | Recommendation |
|---------|----------|----------|----------------|
| Keyboard and screen-reader affordances remain too limited in the SPA | High | `web/static/index.html` | Add labels, landmarks, focus management, and stronger semantic structure |
| Validation and error feedback is still inconsistent across views | Medium | `web/static/index.html` | Standardise inline validation and visible error states |
| Long single-file UI structure makes consistent UX polish harder | Medium | `web/static/index.html:1-6183` | Break the SPA into smaller components/assets to reduce regressions |
| Loading/empty states remain uneven | Low | `web/static/index.html` | Add consistent empty/loading skeletons and button busy states |
| Some confirmation and destructive-action patterns remain implicit | Low | `web/static/index.html` | Make destructive actions consistently explicit and reversible where possible |

## Verdict

The recent CLI improvements slightly improve usability around epic inspection, but the overall UX score still dipped because the main accessibility and UI maintainability issues remain unresolved. The product is usable, but the web surface still relies too heavily on one large file with inconsistent feedback patterns.

## Changes since last assessment
- CLI ticket inspection improved with child counts and clearer child-state emphasis
- The main UX debt remains concentrated in the SPA’s accessibility and consistency gaps

## Remaining recommendations
| Finding | Severity | Recommendation |
|---------|----------|----------------|
| Weak accessibility semantics | High | Add labels, landmarks, keyboard focus management, and ARIA where needed |
| Inconsistent validation/error feedback | Medium | Standardise inline errors and form feedback |
| Oversized SPA file | Medium | Split `web/static/index.html` into smaller maintainable units |
| Uneven loading/empty states | Low | Add consistent busy and empty-state patterns |
| Implicit destructive actions | Low | Add explicit confirmations and clearer consequences |
