# UX Review

**Score: 79/100** (was 78)

## What is being assessed
Interaction quality across the SPA and CLI views: accessibility basics, validation feedback, destructive-action clarity, information density, and whether recent workflow changes improved usability.

## Methodology
Reviewed the main modal markup and keyboard handlers in `web/static/index.html` plus the latest CLI ticket-detail output changes, then refreshed the prior UX report.

## Findings

### Passing checks
- **Modal keyboard handling is now materially better** — focus is trapped on open, restored on close, Escape closes the active modal, and Tab cycles within the modal (`web/static/index.html:5026-5060`).
- **The main modal shells carry explicit dialog semantics** — project, story, ticket, and SDLC modals all use `role="dialog"` / `aria-modal="true"` and heading references (`web/static/index.html:1265-1295`, `web/static/index.html:1394-1416`, `web/static/index.html:1413-1553`).
- **Destructive actions now ask for confirmation instead of silently firing** — delete/archive flows route through `uiConfirm(...)` across the management surfaces (`web/static/index.html:3317`, `web/static/index.html:3595`, `web/static/index.html:3923`, `web/static/index.html:4192`, `web/static/index.html:5864`, `web/static/index.html:6180`).
- **The CLI detail view is easier to scan** — aligned labels and inline child counts reduce visual noise in `tk get` output (`cmd/tk/printer.go:253-290`).

### Issues found
| Finding | Severity | Location | Recommendation |
|---------|----------|----------|----------------|
| Agent, role, and team modals still rely on generic labels instead of the heading-linked pattern used elsewhere | Medium | `web/static/index.html:1303-1349` | Give each modal a heading ID and switch to `aria-labelledby` for consistent screen-reader behavior. |
| The web health control still exposes a 0-4 scale while the CLI health model now scores 10 checks | Medium | `web/static/index.html:1495-1503`, `cmd/tk/cmd_ticket_health.go:218-245` | Align the visual model and copy so web users are not looking at an outdated health scale. |
| The SPA is still one very large file | Medium | `web/static/index.html:1-6287` | Split the UI into smaller components/assets so accessibility and interaction fixes are easier to land consistently. |

## Verdict
UX improved because the recent keyboard and confirmation work is real, and the CLI detail output is noticeably more legible. The remaining UX debt is now mostly consistency debt inside the monolithic SPA rather than total absence of accessibility affordances.

## Changes since last assessment
- Credited focus trapping, Escape handling, and focus restoration as live code rather than open recommendations (`web/static/index.html:5026-5060`).
- Credited the aligned `tk get` detail layout as a real usability improvement (`cmd/tk/printer.go:253-290`).

## Remaining recommendations
| Finding | Severity | Recommendation |
|---------|----------|----------------|
| Inconsistent modal labelling | Medium | Use `aria-labelledby` with real heading IDs across all management modals. |
| Outdated web health scale | Medium | Update the SPA to reflect the 10-check health model or label the control as manual-only. |
| Monolithic SPA file | Medium | Split the UI into smaller maintainable components/assets. |
