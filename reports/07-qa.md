# QA

**Score: 75/100** (was 79)

## What is being assessed

The automated test suite covering ticket/tk: unit and integration volume, package coverage, isolation patterns, CLI/API regression coverage, flakiness risk, and whether the enforced coverage gates still reflect reality.

## Methodology

Counted test functions and suites, ran `go test ./cmd/tk/...`, inspected `cmd/tk/main_test.go`, `internal/server/api_test.go`, `libticket/contract_test.go`, `Makefile`, and the Playwright suite, then compared the current test surface with the previous QA report.

## Findings

### Passing checks
- **Recent CLI regressions now have direct tests** — `tk ls` open-child visibility and done-stage filtering are covered in `cmd/tk/main_test.go:1441-1495`
- **`tk get` child counts are covered** — mixed open/closed child totals verified in `cmd/tk/main_test.go:1855-1880`
- **Child state styling is tested** — child row coloring by lifecycle state is covered in `cmd/tk/main_test.go:4193-4203`
- **The CLI package test suite still passes** — `go test ./cmd/tk/...` succeeds after the current changes
- **Playwright browser coverage remains in place** — 11 specs still exercise the main user-facing flows under `tests/playwright/`
- **Contract testing remains broad** — shared local/HTTP service coverage still lives in `libticket/contract_test.go`

### Issues found
| Finding | Severity | Location | Recommendation |
|---------|----------|----------|----------------|
| Coverage gate remains broken for core packages | High | `Makefile`, `cmd/tk/`, `internal/store/`, `internal/server/` | Raise coverage with focused tests or temporarily lower thresholds until the suite matches the enforced gate |
| No API-level test for stage-role endpoints | Medium | `internal/server/api_test.go` | Add a dedicated `TestSdlcStageRoleAPI` covering add/remove/reorder |
| No CLI tests for `sdlc stage-role-add`, `stage-role-rm`, `stage-role-order` | Medium | `cmd/tk/main_test.go` | Add end-to-end CLI tests for the stage-role command family |
| No CLI coverage for `project set-draft` | Medium | `cmd/tk/main_test.go` | Add a regression test for the command and its visible state change |
| `internal/tui` remains untested | Low | `internal/tui/` | Add basic BubbleTea model/update tests |
| No fuzz targets for parser/input-heavy paths | Low | repository-wide | Add fuzz tests for ref parsing, CLI input normalization, and request decoding |
| `-race` is still absent from the main Go test target | Low | `Makefile` | Add a race-enabled CI target or enable `-race` on `test-go` where practical |

## Verdict

QA regressed despite the new CLI regression tests. The suite is still adding useful coverage around recent command behavior, but the broader quality gate remains in a failing state and some important SDLC command surfaces still lack API/CLI-level tests. The core problem is not test absence everywhere; it is that the enforced thresholds and the actual coverage profile are still out of sync.
