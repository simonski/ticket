# QA / Testing

**Score: 74/100**

## What is being assessed
Test count and coverage, test isolation (t.TempDir, t.Cleanup), mock quality, timing patterns, integration vs unit split, contract test completeness, flakiness risk, and CI enforcement of coverage thresholds.

## Methodology
Counted test files and functions across all packages. Reviewed `libtickettest/contract.go`, `internal/server/api_test.go`, `cmd/ticket/main_test.go`, and all `internal/store/*_test.go` files. Checked for `t.Parallel()`, `t.TempDir()`, `t.Cleanup()`, `time.Sleep()`, and mock patterns. Reviewed `playwright.config.js` and all 11 spec files.

## Findings

### Passing checks
- 390 Go test functions across 31 test files — comprehensive breadth
- 111 Playwright E2E tests across 11 spec files covering auth, tickets, management, chat, hierarchy, workflows
- `libtickettest/contract.go` (1505 lines, 28 operations) runs the same suite against both `LocalService` and `HTTPService` — excellent contract coverage
- `t.TempDir()` used in 82+ tests — proper ephemeral database per test
- `testHandler(t)` in `api_test.go:2291` creates fresh SQLite DB per test via `t.TempDir()`
- No `time.Sleep()` in Go tests — all assertions are immediate or event-driven
- 38 HTTP error status tests in `api_test.go` covering 401, 403, 404
- Per-package coverage thresholds enforced: cmd 55%, libticket 65%, libtickethttp 75%, store 70%, config 70%
- `atomic.Int32` used correctly for thread-safe counters in test assertions (`main_test.go:854`)
- Error path coverage: disabled user login, non-admin access, deleted resource access all tested

### Issues found
| Finding | Severity | Location | Recommendation |
|---------|----------|----------|----------------|
| `playwright.config.js` uses `reuseExistingServer: true` — server state shared between test runs | High | `playwright.config.js:11` | Set `reuseExistingServer: false` in CI; use fresh server per run |
| 585 `wait`/`pause` operations across Playwright tests — high flakiness risk | High | `tests/playwright/` | Audit timing waits; replace with `waitForSelector` or `waitForResponse` |
| No `t.Parallel()` anywhere — sequential execution, missed race detection | Medium | All test files | Add `t.Parallel()` to independent unit tests; run with `-race` flag in CI |
| Test helper functions lack `t.Helper()` | Medium | `api_test.go`, `store_test.go` | Add `t.Helper()` as first line of `testHandler()`, `testDB()`, `assertTableExists()` etc. |
| Coverage thresholds not enforced in CI — only locally via `make test-go-cover` | Medium | `.github/workflows/makefile.yaml` | Replace `make test` with `make test-go-cover` in CI workflow |
| No mocking framework — manual httptest servers, limited mock configurability | Medium | All tests | Consider `testify/mock` for interface mocking; reduces boilerplate |
| Hardcoded test data (usernames "alice", "bob", passwords "password") throughout | Low | `api_test.go` | Extract to constants or test data builder; reduces maintenance burden |
| Most tests use `defer db.Close()` instead of `t.Cleanup()` | Low | `internal/store/*_test.go` | Prefer `t.Cleanup(func() { db.Close() })` for parallel-test safety |

## Verdict
Solid test suite with excellent contract testing, strong error path coverage, and good isolation practices. The main risks are Playwright flakiness from timing-based waits, and coverage thresholds not being enforced in CI (a regression could go undetected). Adding `t.Parallel()` and `-race` would surface concurrency issues that are currently invisible.
