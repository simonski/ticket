# QA

**Score: 77/100** (was 75)

## What is being assessed
Automated regression coverage across CLI, API, browser, and supporting packages, with emphasis on whether recent feature work landed with tests and whether the default quality gates still reflect real risk areas.

## Methodology
Reviewed `Makefile`, `cmd/tk/main_test.go`, `internal/server/api_test.go`, `internal/tui/model_test.go`, and the Playwright suite, then compared the live test surface to the last QA report.

## Findings

### Passing checks
- **Recent CLI workflow changes arrived with direct regression tests** — `tk skill`, aligned `tk get`, and bulk `draft` / `undraft` all have command-level coverage (`cmd/tk/main_test.go:404-452`, `cmd/tk/main_test.go:2167-2201`, `cmd/tk/main_test.go:2235-2270`).
- **The SDLC stage-role API surface is explicitly exercised** — add/reorder/remove flows are covered in one focused API test (`internal/server/api_test.go:726-820`).
- **The quality gates are real, not aspirational** — `make test-go-cover` enforces per-package thresholds and `make test-go-race` exists as a dedicated target (`Makefile:95-125`).
- **Browser regression coverage remains broad** — the repository still carries 11 Playwright specs for auth, navigation, hierarchy, tickets, projects, workflows, and chat (`tests/playwright/*.spec.*`).

### Issues found
| Finding | Severity | Location | Recommendation |
|---------|----------|----------|----------------|
| The expanded `tk health` command has no matching command-level regression test for its new 10-field output or `execute` mode | Medium | `cmd/tk/cmd_ticket_health.go:18-158`, `cmd/tk/main_test.go:1-5952` | Add CLI tests that lock down both single-ticket and `execute` output after the 10-check expansion. |
| The default test flow still omits the race detector | Low | `Makefile:90-96`, `.github/workflows/makefile.yaml:25-45` | Run `make test-go-race` in CI on a scheduled or pre-merge path so concurrency regressions are caught automatically. |

## Verdict
QA is stronger than the previous report claimed because the latest user-visible command work did land with direct tests, and the older stage-role/API gaps are closed. The next gain is to catch up on the expanded health command and to make race detection a normal part of automation rather than an optional manual step.
