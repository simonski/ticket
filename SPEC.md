# Ticket System Remodel Spec

## Status

This document defines the current authoritative model for the remodeled
ticketing system.

It is intentionally opinionated. Where older code or compatibility layers use
task-centric naming, this document takes precedence for the intended external
contract.

## Goals

The system is a ticket management tool for software engineering work.

The remodel has five primary goals:

1. Introduce `project` as a first-class entity.
2. Give every ticket a stable project-scoped human identifier.
3. Preserve the existing `stage` + `state` lifecycle model.
4. Make hierarchy and assignment rules explicit.
5. Provide a CLI and API that are simple enough to use directly from the
   terminal without hidden workflow state.

## Non-Goals

This remodel does not attempt to:

- add sprint planning
- add story points beyond the existing priority/estimate fields
- add cross-project ticket hierarchy
- support arbitrary ticket-type plugins

## Core Entities

There are three primary entities:

1. `project`
2. `ticket`
3. `ticket_history`

`project` is not a ticket type.

## Project Model

`project` is the top-level namespace and container for tickets.

Fields:

- `id`
- `prefix`
- `title`
- `description`
- `acceptance_criteria`
- `notes`
- `status`
- `created_at`
- `updated_at`
- `created_by`

Field rules:

- `id` is an internal immutable identifier. The current implementation uses an
  integer database identifier.
- `prefix` is required, unique, uppercase, and 3 to 5 ASCII letters.
- `title` is required and unique.
- `description` is optional.
- `acceptance_criteria` is optional.
- `notes` is optional.
- `status` is `open | closed`.

Project semantics:

- a closed project does not accept new tickets or lifecycle mutations unless an
  administrator explicitly reopens it
- tickets cannot move between projects

## Ticket Model

`ticket` is the work artifact.

Fields:

- `id`
- `key`
- `project_id`
- `parent_id`
- `type`
- `title`
- `description`
- `acceptance_criteria`
- `priority`
- `stage`
- `state`
- `assignee`
- `created_at`
- `updated_at`
- `created_by`
- `archived`

Field rules:

- `id` is an internal immutable identifier. The current implementation uses an
  integer database identifier.
- `key` is the canonical human identifier.
- `project_id` is required.
- `parent_id` is nullable.
- `title` is required.
- `description` is optional.
- `acceptance_criteria` is optional.
- `priority` is an integer. Initial default remains `1`.
- `assignee` is nullable.
- `archived` is boolean.

`key` format:

- `{PROJECT_PREFIX}-{TYPE_CODE}-{SEQUENCE}`

Examples:

- `CUS-E-12`
- `CUS-T-143`
- `OPS-B-9`

`key` rules:

- unique across the whole system
- generated by the server
- immutable after creation
- sequence is monotonically increasing per project

This spec does not use truncated UUIDs or shortened ULIDs for human ticket
keys. Short opaque IDs are less readable and still need collision management.
A project-local sequence is clearer, collision-free, and easier to use in the
CLI and UI.

## Ticket Types

Supported ticket types:

- `epic`
- `task`
- `bug`
- `spike`
- `chore`

Type codes for keys:

- `E` = `epic`
- `T` = `task`
- `B` = `bug`
- `S` = `spike`
- `C` = `chore`

`design` is not a ticket type. It is a lifecycle stage only.

## Ticket Hierarchy

Hierarchy is represented by `parent_id`.

Allowed parent-child relationships:

- `epic` may parent `epic | task | bug | spike | chore`
- `task` may parent `task | bug | spike | chore`
- `bug` may not parent any ticket
- `spike` may not parent any ticket
- `chore` may not parent any ticket

Additional hierarchy rules:

- parent and child must belong to the same project
- a ticket may have at most one parent
- cycles are forbidden
- a parent ticket with children is a derived ticket for lifecycle purposes

Terminology:

- leaf ticket: a ticket with no children
- parent ticket: a ticket with one or more children

## Lifecycle Model

The existing lifecycle model remains the basis of ticket progress.

### Stage

Allowed stages:

- `design`
- `develop`
- `test`
- `done`

Meaning:

- `design`: the work is being appraised, explored, or refined
- `develop`: implementation is being done
- `test`: the outcome is being verified
- `done`: the ticket is complete and closed

### State

Allowed states:

- `idle`
- `active`
- `complete`

Meaning:

- `idle`: ready but not currently in progress
- `active`: currently being worked on with a named assignee
- `complete`: work for the current stage is complete

Valid combinations:

- `design`: `idle | active | complete`
- `develop`: `idle | active | complete`
- `test`: `idle | active | complete`
- `done`: `complete`

### Rendered Status

`status` is not stored independently.

It is always rendered as:

- `<stage>/<state>`

Examples:

- `design/idle`
- `develop/active`
- `done/complete`

### Lifecycle Invariants

- `state=active` requires `assignee != ""`
- `state=idle` may be unassigned
- `state=complete` may retain assignee
- `stage=done` requires `state=complete`
- `stage!=done` allows `idle | active | complete`

## Lifecycle Mutation Rules

Only leaf tickets may be directly changed by stage/state commands.

If a ticket has children, direct lifecycle mutation must fail with:

```text
ticket has children; stage/state is derived from descendants
```

### Ticket Creation Defaults

On creation of a non-project ticket:

- `stage=design`
- `state=idle`

### Stage Commands

These mutate leaf tickets only:

- `ticket design`
- `ticket develop`
- `ticket test`
- `ticket done`

Behavior:

- `design` sets `stage=design`, `state=idle`
- `develop` sets `stage=develop`, `state=idle`
- `test` sets `stage=test`, `state=idle`
- `done` sets `stage=done`, `state=complete`

### State Commands

These mutate leaf tickets only:

- `ticket idle`
- `ticket active`
- `ticket complete`

Behavior:

- `idle` keeps stage and sets `state=idle`
- `active` keeps stage and sets `state=active`
- `complete` keeps stage and sets `state=complete`

## Derived Parent Lifecycle

If a ticket has children, its effective lifecycle is derived recursively from
its descendants.

### Effective Parent Stage

Parent effective stage is the earliest stage among all descendants.

Stage ordering:

- `design < develop < test < done`

### Effective Parent State

Parent effective state is derived as:

- `complete` if all descendants are complete
- `active` if any descendant is active
- `idle` otherwise

Implications:

- when any leaf moves to `active`, all ancestors become effectively `active`
- a parent reaches `done` only when all descendants are `done/complete`

## Assignment Rules

Assignment attaches a named user to a ticket.

Rules:

- only tickets may be assigned
- assignment is allowed only within the same project namespace
- parent tickets may be assigned, but only leaf tickets may be claimed
- `active` requires an assignee
- `unassign` is forbidden if the resulting ticket would remain `active`

Recommended server behavior:

- if a ticket is set to `active` without an assignee, reject the request
- if a ticket is unassigned while `active`, force the caller to move it out of
  `active` first

## Claim Rules

Claim is a server-mediated assignment workflow using the authenticated user.

`claim` never accepts a username flag. The actor is always the current session
user.

Eligibility rules:

- only leaf tickets are claimable
- only tickets in `develop/idle` are claimable
- only tickets in open projects are claimable
- archived tickets are not claimable

Claim behavior:

- `ticket claim -id <KEY>`:
  - if the ticket is eligible, set `assignee=current_user` and `state=active`
  - if the ticket is not eligible, return a validation error
- `ticket claim`:
  - if the current user already has a claimed active leaf ticket, return that
    ticket
  - otherwise, assign one eligible ticket and return it
- `ticket claim -dry-run`:
  - evaluate the same selection logic as `ticket claim`
  - return the candidate without changing server state

Default server-side selection order for `ticket claim` without `-id`:

1. highest priority first
2. oldest created ticket first
3. key lexical order as final tie-break

This selection rule must be documented and tested so claim behavior is
predictable.

## History Model

History is stored separately from the ticket row.

`ticket_history` fields:

- `id`
- `ticket_id`
- `project_id`
- `actor`
- `event_type`
- `from_value`
- `to_value`
- `payload`
- `created_at`

History is append-only.

Representative event types:

- `ticket_created`
- `ticket_updated`
- `ticket_attached`
- `ticket_detached`
- `ticket_assigned`
- `ticket_unassigned`
- `ticket_claimed`
- `ticket_stage_changed`
- `ticket_state_changed`
- `ticket_parent_stage_changed`
- `ticket_parent_state_changed`
- `comment_added`

Derived lifecycle updates for parent tickets must also emit history entries.

## CLI Contract

`project` and `ticket` are separate command groups within the `ticket` binary.

`project` commands:

- `ticket project create -prefix <PREFIX> "TITLE"`
- `ticket project list`
- `ticket project get <PREFIX|PROJECT_ID>`
- `ticket project <PREFIX|PROJECT_ID> update`
- `ticket project <PREFIX|PROJECT_ID> disable`
- `ticket project <PREFIX|PROJECT_ID> enable`

`ticket` commands:

- `ticket create -project <PREFIX|PROJECT_ID> -type <TYPE>`
- `ticket list`
- `ticket get <KEY|TICKET_ID>`
- `ticket update <KEY|TICKET_ID>`
- `ticket delete <KEY|TICKET_ID>`
- `ticket attach <KEY|TICKET_ID> <KEY|TICKET_ID>`
- `ticket detach <KEY|TICKET_ID>`
- `ticket assign <KEY|TICKET_ID> <USERNAME>`
- `ticket unassign <KEY|TICKET_ID> <USERNAME>`
- `ticket claim`
- `ticket claim <KEY|TICKET_ID>`
- `ticket claim -id <KEY|TICKET_ID>`
- `ticket claim -dry-run`
- `ticket unclaim <KEY|TICKET_ID>`
- `ticket design <KEY|TICKET_ID>`
- `ticket develop <KEY|TICKET_ID>`
- `ticket test <KEY|TICKET_ID>`
- `ticket done <KEY|TICKET_ID>`
- `ticket idle <KEY|TICKET_ID>`
- `ticket active <KEY|TICKET_ID>`
- `ticket complete <KEY|TICKET_ID>`

CLI notes:

- `project` is not passed through `ticket -type`
- `detach` does not require `-parent`; the current parent is implicit
- `unassign` remains an admin operation and currently accepts a username
- list/get commands should accept either human keys or internal IDs where
  practical, but human keys are preferred in output

## API Contract

The API must expose:

- project CRUD and list operations
- ticket CRUD and list operations
- attach and detach operations
- assign, unassign, and claim operations
- explicit stage and state transitions
- effective lifecycle values for parent tickets
- append-only history reads

Current route notes:

- `/api/tickets`, `/api/tickets/...`, and `/api/projects/{id}/tickets` are the
  preferred ticket-first routes
- the old `/api/tasks` route family has been removed from the supported
  external contract

API responses should include:

- internal `id`
- human `key` for tickets
- effective `stage`
- effective `state`
- rendered `status`

For parent tickets, the API should return effective lifecycle values only.

Minimum schema additions or changes:

### Projects

Projects table must include:

- `id`
- `prefix`
- `title`
- `description`
- `acceptance_criteria`
- `notes`
- `status`
- `created_at`
- `updated_at`
- `created_by`

Constraints:

- unique `prefix`
- unique `title`
- `prefix` format validation
- `status in ('open','closed')`

### Tickets

Tickets table must include:

- `id`
- `key`
- `project_id`
- `parent_id`
- `type`
- `title`
- `description`
- `acceptance_criteria`
- `priority`
- `stage`
- `state`
- `assignee`
- `created_at`
- `updated_at`
- `created_by`
- `archived`

Constraints:

- unique `key`
- `type in ('epic','task','bug','spike','chore')`
- `stage in ('design','develop','test','done')`
- `state in ('idle','active','complete')`
- reject `stage='done'` with non-`complete` state

Ticket history table must include:

- `id`
- `ticket_id`
- `project_id`
- `created_by`
- `event_type`
- `payload`
- `created_at`

## Migration Status

The main remodel phases are complete:

- project prefixes and human ticket keys are implemented
- `spike` and `chore` are supported ticket types
- the CLI and docs are project/ticket oriented
- `ticket_history` is present and used for append-only reads
- claim ordering and leaf-only claim behavior are implemented

## Test Requirements

Tests must cover:

- project prefix validation and uniqueness
- human ticket key generation and per-project sequence behavior
- hierarchy validation, including cycle rejection
- same-project parent/child enforcement
- lifecycle mutation rules for leaf tickets
- derived lifecycle recalculation for parent tickets
- assignment and unassignment validation
- claim eligibility and selection ordering
- history emission for direct and derived changes
- CLI parsing for `project` and `ticket` command separation
- API serialization of `id`, `key`, `stage`, `state`, and `status`

## Final Decisions

1. Project prefixes are 3 to 5 uppercase ASCII letters.
2. Project titles remain globally unique.
3. Closed projects do not allow routine ticket mutation; reopen first.
4. `spike` and `chore` are not allowed as parents.

## Summary

The remodel should converge on this shape:

- `project` is a first-class entity and separate CLI/API resource
- tickets belong to exactly one project
- tickets use readable project-scoped keys such as `CUS-T-143`
- internal identifiers remain stable implementation-level IDs
- lifecycle remains `stage + state`
- parent ticket lifecycle remains derived from descendants
- claim is explicit, authenticated, and deterministic
- history is append-only and normalized
